# MLパイプライン修正フロー

Planner/Generator に加え、Evaluator を2軸(Spec/Standards)に分けてレビューする。
フック制御用の環境変数(CLAUDE_WORK_SCOPE / CLAUDE_ENFORCE_EVAL / CLAUDE_EVAL_CMD)は
`.claude/settings.local.json` の `env` ブロックに設定する
(README「起動(フックの有効化)」の節を参照)。

対象に設計書(docs/active/)がある場合、`CLAUDE_SPEC_CHECK=1` を設定すると
Stop フックの `spec_gate` が「## 受け入れ条件」テーブルの全要件PASS・承認・監査OKを
機械検査し、欠けがあれば完了をブロックする。manual要件はユーザーが
`! uv run python .claude/hooks/spec_approve.py <要件ID>` で承認するまで通らない。
設計書自体もユーザーの計画承認(上記の要件承認時に自動記録、または
`! uv run python .claude/hooks/spec_approve.py --design <設計書名>`)が必要で、
承認時点のハッシュと照合されるため、承認後に設計書を書き換えるとブロックされる
(設計書を正当に更新した場合はユーザーに再承認を依頼する)。
spec_approve.py はユーザーの `!` 実行専用で、エージェントの Bash 経由の実行は
guard_bash がブロックする。

## 前提: 作業スコープの確定
- $ARGUMENTS の冒頭で作業ディレクトリ(例: projects/Deep_MIL)が指定されていれば、それを作業スコープとする。
- 指定がなければ、着手前にユーザーに「どのプロジェクトディレクトリで作業するか」を必ず確認する。
- 各エージェントへのタスク指示には作業スコープを必ず明記する。
  `git diff` / `git add` は作業スコープにパスを限定させる
  (例: `git diff -- projects/Deep_MIL/`、`git add projects/Deep_MIL/`)。

## 手順
### 1. 作業スコープの確定
作業スコープを確定する(上記参照)。

### 1.5. 作業ブランチの作成
現在のブランチを記録してから、作業用ブランチを作成してチェックアウトする。

```
git checkout -b pipeline/YYYYMMDD-<トピック>
```

ブランチ名の <トピック> は、ユーザーの依頼内容から2〜3語の英語スラグを
生成する(例: `pipeline/20260722-fix-attention-viz`)。

以降、全ての git add / git commit はこのブランチ上で行う。
main ブランチには一切コミットしない。

### 2. 事前調査
- 作業スコープ直下に CONTEXT.md があればここで一度だけ読み、用語の要点を
  以降の各エージェントへのタスク指示に含める(各エージェントに再読させない)。
- 対象コードの範囲が広い場合は、planner を起動する前に Explore サブエージェント
  (Haiku)で大まかな下調べを行い、その要約を planner への指示に含める
  (Opus の調査トークンを節約するため)。

### 3. 計画の作成
planner に「やりたいこと」+ CONTEXT要点 + Explore要約を渡し、作業スコープ内を
調査させて実装計画を作成させる。計画には並列化判定(並列化可能 / 逐次のみ)を含める。

### 3.5. 計画の承認判定
環境変数 CLAUDE_AUTO_APPROVE の値で動作が変わる。

**CLAUDE_AUTO_APPROVE=1 の場合:**
plan-reviewer に計画を審査させる。
- plan-reviewer が「自動承認OK」と判定 → ユーザー承認をスキップして手順5(実装)へ
- plan-reviewer が「自動承認NG」と判定 → ユーザーに計画を提示し承認を待つ

**CLAUDE_AUTO_APPROVE=0 または未設定(デフォルト)の場合:**
従来通りユーザーに計画を提示し、承認されるまで進めない。

### 4. 計画の承認(手順3.5でスキップされなかった場合)
計画をユーザーに提示し、承認されるまで進めない。

ユーザーが計画を却下した場合、.claude/improvements/feedback.md に
以下を追記してから計画を修正する
(.claude/improvements/ が無ければ作成する)。

```
## YYYY-MM-DD [planner: 計画却下]
- 却下の理由: (ユーザーの発言を要約)
```

### 5. 実装
計画の並列化判定を確認する。

**「並列化可能」かつ tmux 環境がある場合(並列実装):**
統合ブランチ(pipeline/YYYYMMDD-<トピック>)から、並列グループごとに
サブブランチを作成する。

```
git checkout -b pipeline/YYYYMMDD-<トピック>-group-A
git checkout pipeline/YYYYMMDD-<トピック>
git checkout -b pipeline/YYYYMMDD-<トピック>-group-B
```

各チームメイトは自分のサブブランチ上でのみコミットする。
他グループのファイルには一切触らない。
各チームメイトが実装完了したら手順5.5へ進む。

**「逐次のみ」の場合、または tmux が使えない場合(逐次実装):**
統合ブランチ(pipeline/YYYYMMDD-<トピック>)上で
従来通り generator に逐次実装させる。サブブランチは作らない。
tmux 環境が無い場合に「並列化可能」な計画を逐次実行する際は、
「並列化可能な計画ですが、tmux環境が無いため逐次で実行します」と報告する。

generator には計画ファイルのパスを渡して実装させ、完了報告から変更ファイルの一覧を受け取る。

### 5.5. Codex クロスレビュー(条件分岐)
環境変数 CLAUDE_CROSS_REVIEW の値で動作が変わる。
- `1`: cross-review スキルを実行する(codex_gate.py が未実行時にブロックする)
- `0` または未設定: スキップする

cross-review スキルが実行された場合、そのレポートを手順6の
evaluator / evaluator-standards に「Codexからの追加指摘」として渡す。

### 6. レビュー
実装完了後、以下2つを独立に実行する(互いの判断が影響し合わないよう、
それぞれ別の視点でレビューさせる)。両者には変更ファイル一覧を渡し、
diff の確認範囲をそのパスに限定させる。

- evaluator: 計画通りに動作するか(Spec)。評価スクリプトを実際に実行し数値で判定。
- evaluator-standards: コーディング規約・可読性・型安全性・コードスメル(Standards)。

並列実装かつ tmux 環境がある場合、evaluator と evaluator-standards も
チームメイトとして並列実行する。

### 6.5. 原子性チェック(並列実装の場合のみ)
並列グループごとにレビュー結果を確認する。

**全グループが PASS の場合:**
各サブブランチを統合ブランチにマージする。

```
git checkout pipeline/YYYYMMDD-<トピック>
git merge pipeline/YYYYMMDD-<トピック>-group-A --no-ff
git merge pipeline/YYYYMMDD-<トピック>-group-B --no-ff
```

マージコンフリクトが発生した場合はユーザーに報告し判断を仰ぐ。
統合ブランチ上で統合テスト(全テスト実行)を行い、通れば手順7へ進む。

**1つでも NEEDS_REVISION の場合:**
他のグループが PASS していても、main へのマージはしない(原子性の保証)。
失敗したグループのサブブランチだけを修正対象として generator に差し戻す。
PASS済みのグループはそのまま保持する。
修正後、失敗グループだけ再レビューする。

**1つでも FAIL(3回連続)の場合:**
全サブブランチの統合を中止する。
ユーザーに報告し、以下の選択肢を提示する。
- 全ブランチを破棄して Planner からやり直す
- 失敗グループだけ破棄し、PASS済みグループは保持して部分的に進める

### 6.7. リファクタリング・パス(磨きの工程)
evaluator と evaluator-standards が両方 PASS した場合のみ実行する。

**前提**: 開始時点で作業ツリーはクリーンであること(実装は手順5で全て
コミット済みのはず)。クリーンでない場合は磨きを行わず手順7へ進む
(後述の復元が「磨きの取り消し」と正確に一致することを保証するため)。

generator に以下を指示して、動作を一切変えずにコードを磨く1周を行わせる。

**リファクタリングの観点(この順に確認):**
1. **音読テスト**: コードを上から日本語で音読して、自然な文章として
   説明できるか。読めない箇所は、読める形に書き換える。
   - 読める例: `if not is_valid_bag: return None`
     →「有効なバッグでなければ、何も返さない」
   - 読めない例: `if not (b.sz > 0 and b.id in kn): return None`
2. **命名の磨き**: 変数名・関数名を「この文脈でもっと正確な名前」に
   できないか見直す。
3. **コメントの解消**: 処理の説明コメントが必要な箇所を、コメント不要な
   コード(意味の分かる変数への切り出し等)に書き換えられないか。
   why コメントは残す。what コメントはコード化して消す。
4. **簡潔化**: 3行で書けることを5行で書いていないか。
5. **対称性**: 似た処理は似た形になっているか。

**制約:**
- 動作を一切変えない(テストの結果が1つでも変わったら失敗)
- 新機能・新抽象化を追加しない(これはリファクタであって拡張ではない)
- 既存ファイルの編集のみ。**新規ファイルを作らない**(復元を単純に保つため)
- 変更は磨き対象のファイルのみ。スコープを広げない
- 1周だけ。磨きのループを回さない

**リファクタ後(コミット前に検証する):**
- 計画の検証コマンドと全テストを再実行する(Spec軸の再確認)
- evaluator-standards に**磨きの未コミット差分(`git diff HEAD`。staged 込み)**だけを渡して
  再確認させる(Standards軸の再確認。磨きで命名・構造が変わるため、
  変更前の PASS 判定をそのまま流用しない)
- 全て PASS → refactor コミット(`refactor(step N): <要約>`。
  N は磨き対象の計画ステップ番号、全体の磨きなら計画の最終ステップ番号)をして手順7へ
- 1つでも FAIL → `git checkout HEAD -- <磨きで触ったファイル>` で復元し
  (staged 済みの変更も含めて戻る)、
  `git status` に未追跡ファイルが現れていれば削除して(制約違反の後始末)、
  手順7へ進む(磨きの失敗で全体を止めない。開始時クリーンが前提なので、
  この復元は「磨きの取り消し」と正確に一致する)

### 7. 結果の集約
- evaluator が PASS かつ evaluator-standards が PASS: 手順7.5へ
- どちらかが NEEDS_REVISION: 指摘をまとめて generator に差し戻す(手順5に戻る)
- evaluator が FAIL(3回連続): planner からやり直すことを提案(手順3に戻る)

差し戻し時のトークン節約(必ず守る):
- 差し戻しは**新規の generator エージェント**に指摘の全文+対象ファイル一覧を
  渡して行う。前回の generator を会話再開で呼び戻さない(全履歴を引きずって
  コンテキストが数十万トークン規模に膨らむ。指摘全文があれば新規で十分)。
- 再レビューは **NEEDS_REVISION を出した軸だけ**に依頼する。PASS 済みの軸は、
  差し戻し修正がその軸の観点(Spec なら動作・要求充足、Standards なら規約・
  可読性)に触れるファイル変更を含む場合のみ再実行する。

### 7.5. spec-auditor による独立監査(条件分岐)
対象に設計書(docs/active/)があり受け入れ条件テーブルがある場合、evaluator ×2 の後に
spec-auditor を起動し、verdict ファイルの証拠を独立コンテキストで再検証させる。
spec-auditor が NG を出した要件があれば generator に差し戻す。全件 OK なら完了レポートを出力する。

### 8. イテレーション制限
最大3イテレーションで収束しなければユーザーに判断を仰ぐ。

### 9. マージ確認
全工程が完了したら、ユーザーに以下を提示する。

- 作業ブランチ名: pipeline/YYYYMMDD-<トピック>
- コミット数: N件
- 変更ファイル数: `git diff --stat main...<ブランチ>` の結果
- 並列化した場合: 各グループのマージ状態
- 全変更の要約: 1〜3文

そして「mainにマージしますか?」と確認する。

- **「はい」**: `git checkout main && git merge <統合ブランチ> --no-ff` を
  実行し、マージ完了を報告する。`--no-ff` でマージコミットを残し、
  パイプライン実行の単位が履歴で分かるようにする。
  マージ後、サブブランチを削除する。
- **「diffを見せて」**: `git diff main...<統合ブランチ>` を表示する。
  確認後、改めてマージするか聞く。
- **「いいえ」**: ブランチをそのまま残して終了する。
  「後でマージしたい場合は `git checkout main && git merge <ブランチ>`
  を実行してください」と案内する。
- **「全部捨てて」**: 統合ブランチとサブブランチを全削除し、main に戻る。
  main は無傷。

$ARGUMENTS を「作業スコープ + やりたいこと」として解釈する。
