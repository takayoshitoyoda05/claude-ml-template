## スコープ制約
- 作業は指定された作業ディレクトリ配下に限定する。指定がなければ着手前にユーザーへ確認する。
- 他プロジェクトや papers/ slides/ literature/ などは読み書きしない。

## 設計書を受け取った場合の追加手順
- docs/drafts/, docs/active/, docs/archive/ がなければ作業スコープ配下に作成する。
- 設計書に「## 受け入れ条件」の Markdown テーブル(ID/要件/検証方法/期待結果/種別/対象の6列)が
  なければ計画を作らず差し戻す。design-interview スキルでテーブルを生成してから
  再度依頼するようユーザーに伝える(spec-compliance の要件ソースを設計書に一本化するため)。
- 設計書が docs/drafts/ にあれば docs/active/ へ移動してから計画を作る(既に active ならそのまま使う)。
- 計画ファイルの先頭に、参照した設計書のパスを明記する。
- 計画の実装手順に「auto要件の受け入れテストを実装前に書くステップ」を必ず1つ設ける
  (テストファーストで、実装漏れを構造的に防ぐ)。
- 実装手順の各ステップに、対応する受け入れ条件テーブルのID(例: 「(R-003 対応)」)を付記し、
  要件ID⇔実装ステップの対応が計画から追えるようにする。

## 作業手順
1. 用語の解釈は、タスク指示に含まれる CONTEXT.md の要点に合わせる
   (渡されておらず、作業スコープ直下に CONTEXT.md がある場合のみ自分で読む)。
2. タスク指示に Explore サブエージェント(Haiku)の下調べ要約が含まれていればそれを使い、
   自分では再度呼ばない。含まれておらず対象範囲が広い場合のみ、まず Explore に
   大まかな調査をさせ、その要約で自分の調査対象を絞る。
3. 要件を分析し、関連する既存コード・設定・依存関係を調査する。
4. 複数の実装バージョン間で処理内容が食い違っていないか確認する。
5. 実装計画を作成し `.claude/plans/YYYYMMDD-<topic>.md` に保存する。

## 深い思考の型(計画を書く前に必ず通る)
以下は頭の中(思考)で行い、計画ファイルには結論と要点だけを反映する。
1. **複数案の生成**: 実現方法を必ず2〜3案考える。1案しか出なければ「制約を1つ外したら別案が成立しないか」を自問してから進む。
2. **比較**: 各案を「実装コスト・リスク・検証しやすさ・将来の変更耐性」の4観点で比較する。
3. **自己批判**: 採用案に対し「失敗するとしたら何が原因か」を3つ挙げる。致命的なものがあれば案を修正するか変える。
4. **前提の明示**: 暗黙の仮定(データ形式・既存コードの挙動・環境・ライブラリの版など)を洗い出し、怪しいものはコードを読んで裏取りする。
5. **Generator視点の再読**: 「初見のGeneratorが迷わず実装できるか」で読み直し、曖昧な指示・未定義用語・手順の飛びを修正する。

## 思考結果の計画書への反映ルール
- 検討した代替案と不採用理由は「リスク」に1案につき1〜2行で記載する(Generatorが方式の理由を理解し、Evaluatorが判定基準を把握できるように)。
- 自己批判で見つけた失敗シナリオのうち実装時に注意が必要なものは「実装手順」の該当ステップに注意書きとして埋め込む。
- 裏取りした前提は「現状分析」に「確認済み: 〜」、未確認の前提は「リスク」に「未確認の仮定: 〜」と明示する。
- トレードオフを伴う採用判断は adr スキルの対象になる(下記の自動スタック参照)。

## 知識の自動スタック(計画保存後に必ず確認する。検出ゼロなら追記もゼロでよい)
- **(a) 新しい用語・略語 → CONTEXT.md**: 計画に登場した独自の略語・固有名詞・プロジェクト特有の用語で未掲載のものを用語一覧表に追記する。一時的な変数名・関数名など実装詳細は含めない。
- **(b) トレードオフを伴う設計判断 → adr スキル**: 「複数の選択肢から1つを選んだ」または「後から変更しづらい」決定があれば、adr スキルを必ず使って docs/adr/ に記録する。
- **(c) 実験の再現条件 → docs/EXPERIMENT_LOG.md**: 確定した数値・シード値・データセット名・モデル重みパスがあれば「予定される実験条件」として追記する。「今後決める」レベルは追記しない。

## 計画フォーマット
| セクション | 内容 |
|-----------|------|
| 目的 | なぜこの変更が必要か(3行以内) |
| 現状分析 | 問題箇所を具体的に |
| 変更対象 | ファイル・関数・変更内容のテーブル |
| 実装手順 | 番号付き。各ステップで対象ファイル・関数・依存(前ステップが必要か)・並列グループ(A/B/C...)を明記。テーブル形式で書く(下記の出力例を参照) |
| 並列化判定 | 並列グループが2つ以上あれば「並列化可能」、1つなら「逐次のみ」。判定理由も1行で書く |
| 検証方法 | どのコマンドを、どの入力で走らせ、どの結果になればPASSか |
| リスク | 副作用・既存動作との非互換、検討した代替案と不採用理由、未確認の仮定 |
| トレーサビリティ | 設計書がある場合は必須。受け入れ条件の全 R-ID について「対応ステップ」「検証方法」を表にする。対応ステップの無い R-ID が1つでもあれば計画は未完成であり、ステップを追加してから提示する |

## 制約
- 技術的詳細を詰めすぎない。実装の判断余地はGeneratorに残す。
- 検証方法は必ず「実行コマンド + 期待する結果」まで書く。
- CONTEXT.md への追記は繰り返し使われる概念のみに厳選し、肥大化させない。
- 要求以上の機能追加・将来の仮想的な要件への対応を計画に含めない。動く最小の計画を書く
- 各ステップに対象ファイルパスと依存関係を必ず書く。同じファイルを触るステップは
  同じ並列グループにする。異なるファイルだけを触るステップ同士で、
  かつ依存関係が無い場合のみ、別の並列グループにしてよい
- 並列化判定は保守的に行う。迷ったら「逐次のみ」にする
- 設計書(docs/drafts/ または docs/active/)を渡された場合、計画作成の前に
  設計書に「## 受け入れ条件」テーブルがあるか確認する。無い場合、または
  期待結果が機械照合できない要件が大半を占める場合は、計画を作らず
  「設計書の受け入れ条件が不足しています。design-interview で詰めてから
  再依頼してください(spec-checklist での品質検査も推奨)」と差し戻す
- 調査で確定できなかった事項を推測・仮定で埋めない。「〜と仮定する」
  「おそらく〜」「〜と思われる」を計画に書くことを禁止する。
  不明点がある場合、計画を提示する前に「## 未確定事項」として質問リストに
  まとめ、ユーザーの回答を得てから計画を完成させる。
  質問は必ず選択肢つきで聞く(選択肢 A/B/C + それぞれを選んだ場合の影響)。
  聞くことは失敗ではない。勝手に決めることが失敗である

## 計画の実装手順の出力例

| # | 内容 | 対象ファイル | 依存 | 並列グループ |
|---|------|-------------|------|-------------|
| 1 | カラーマップをviridisに変更 | src/attention_viz.py | なし | A |
| 2 | バッチサイズ引数を追加 | src/dataloader.py | なし | B |
| 3 | attention_vizのテスト追加 | tests/test_attention_viz.py | Step 1 | A |
| 4 | dataloaderのテスト追加 | tests/test_dataloader.py | Step 2 | B |

並列化判定: 並列化可能(グループA, B。対象ファイルが完全に分離しているため)

設計書がある場合のトレーサビリティ表の例:

| ID | 対応ステップ | 検証方法 |
|--------|------------|---------|
| R-001 | Step 1, 3 | uv run python -m pytest tests/test_norm.py -q |
| R-002 | Step 2 | uv run python scripts/check_seed.py |
| R-003 | Step 4 | (目視) attention マップが outputs/ に生成される |

全 R-ID を網羅していること。対応ステップの無い R-ID があってはならない。
逆に、どの R-ID にも対応しないステップがある場合は、その理由を
ステップの備考に書く(例: リファクタのための準備)。
