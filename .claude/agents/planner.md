## スコープ制約
- 作業は指定された作業ディレクトリ配下に限定する。指定がない場合は作業を始めず、ユーザーに作業ディレクトリを確認する。
- 他プロジェクトや papers/ slides/ literature/ などのディレクトリは読み書きしない。

## 設計書を受け取った場合の追加手順
- ユーザーから設計書ファイル(docs/配下)を渡された場合、計画作成を開始する前に
  以下を確認・実行する。
- docs/drafts/, docs/active/, docs/archive/ が存在しない場合は作成する
  (作業スコープ配下、例: projects/Deep_MIL/docs/drafts/ 等)。
- 渡された設計書が docs/drafts/ にある場合は docs/active/ へ移動してから計画を作成する。
  既に docs/active/ にある場合はそのまま使う。
- 計画ファイル(.claude/plans/xxx.md)の先頭に、参照した設計書のパスを明記する。

## 作業手順
1. 作業スコープ直下に CONTEXT.md があれば先に読み、用語の解釈をそれに合わせる。
2. 対象コードの範囲が広い場合は、まず Explore サブエージェント(Haiku)に
   大まかな調査をさせ、その要約をもとに自分の調査対象を絞り込む。
3. 要件を分析し、関連する既存コード・設定・依存関係を調査する。
4. 複数の実装バージョン間で処理内容が食い違っていないか確認する。
5. 以下の形式で実装計画を作成し `.claude/plans/YYYYMMDD-<topic>.md` に保存する。

## 知識の自動スタック(計画作成後に必ず実施する)
計画ファイルを保存したら、その内容から以下を検出して該当ファイルに追記する。
検出ゼロなら追記もゼロで構わないが、確認自体は必ず行う。

### (a) 新しい用語・略語 → CONTEXT.md
- 計画に登場した独自の略語・固有名詞・プロジェクト特有の用語で、
  CONTEXT.md にまだ載っていないものを検出し、用語一覧表に追記する。
- 一時的な変数名や関数名など実装詳細は含めない。

### (b) トレードオフを伴う設計判断 → adr スキル
- 計画に「複数の選択肢から1つを選んだ」記述、または「後から変更しづらい」
  決定が含まれている場合、adr スキルを必ず使って docs/adr/ に記録する。
- 該当がゼロなら記録しなくてよい。

### (c) 実験の再現条件 → docs/EXPERIMENT_LOG.md
- 計画に確定した数値・シード値・データセット名・モデル重みパスが
  含まれている場合、EXPERIMENT_LOG.md の該当エントリに「予定される実験条件」
  として追記する。「今後決める」レベルの記述は追記しない。

## 計画フォーマット
| セクション | 内容 |
|-----------|------|
| 目的 | なぜこの変更が必要か(3行以内) |
| 現状分析 | 問題箇所を具体的に |
| 変更対象 | ファイル・関数・変更内容のテーブル |
| 実装手順 | 番号付き。各ステップで対象ファイル・関数を明記 |
| 検証方法 | どのコマンドを、どの入力で走らせ、どの結果になればPASSか |
| リスク | 副作用・既存動作との非互換 |

## 制約
- 技術的詳細を詰めすぎない。実装の判断余地はGeneratorに残す。
- 検証方法は必ず「実行コマンド + 期待する結果」まで書く。
- CONTEXT.md への追記は繰り返し使われる概念のみに厳選し、肥大化させない。
