## スコープ制約
- 指示された作業ディレクトリ配下のみを対象にする(スコープ外への書き込みはフックが物理ブロックする)。指定がなければ着手前にユーザーに確認する。

## 設計書を受け取った場合の追加手順
- ユーザーから設計書ファイル(docs/配下)を渡された場合、計画作成を開始する前に
  以下を確認・実行する。
- docs/drafts/, docs/active/, docs/archive/ が存在しない場合は作成する
  (作業スコープ配下、例: projects/Deep_MIL/docs/drafts/ 等)。
- 渡された設計書が docs/drafts/ にある場合は docs/active/ へ移動してから計画を作成する。
  既に docs/active/ にある場合はそのまま使う。
- 計画ファイル(.claude/plans/xxx.md)の先頭に、参照した設計書のパスを明記する。

## 作業手順
1. 用語の解釈は、タスク指示に含まれる CONTEXT.md の要点に合わせる
   (渡されておらず、作業スコープ直下に CONTEXT.md がある場合のみ自分で読む)。
2. 要件を分析し、関連する既存コード・設定・依存関係を調査する。
   タスク指示に事前調査(Explore)の要約が含まれていれば、それを起点に調査範囲を絞る。
3. 複数の実装バージョン間で処理内容が食い違っていないか確認する
4. 以下の形式で実装計画を作成し `.claude/plans/YYYYMMDD-<topic>.md` に保存する
5. 計画にアーキテクチャ上重要な決定(トレードオフを伴う選択)が含まれる場合、
   adr スキルを使って docs/adr/ に記録することをユーザーに提案する。

## 計画フォーマット
| セクション | 内容 |
|-----------|------|
| 目的 | なぜこの変更が必要か(3行以内) |
| 現状分析 | 問題箇所を具体的に |
| 変更対象 | ファイル・関数・変更内容のテーブル |
| 実装手順 | 番号付き。各ステップで対象ファイル・関数を明記 |
| 検証方法 | どのコマンドを、どの入力で走らせ、どの結果になればPASSか |
| リスク | 副作用・既存動作との非互換 |

## 制約
- 技術的詳細を詰めすぎない。実装の判断余地はGeneratorに残す
- 検証方法は必ず「実行コマンド + 期待する結果」まで書く
