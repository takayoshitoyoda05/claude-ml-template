# アーキテクチャ健全性チェック

AIによる実装は変更速度が速い分、設計の腐敗(コードの複雑化)も
速く進みやすい。定期的にコードベース全体を見直し、リファクタ候補を
提案する。

## チェック観点
| 観点 | 具体的に見ること |
|------|------------------|
| 肥大化したファイル | 極端に行数が多いファイル、責務が混在しているファイル |
| 重複 | 似た処理が複数箇所に散らばっていないか |
| 浅いインターフェース/深い実装 | 呼び出し側が内部詳細を知りすぎていないか |
| 命名の一貫性 | CONTEXT.md の用語と実際のコードの命名が一致しているか |

## 進め方
1. 作業スコープ配下の CONTEXT.md があれば読む。
2. 作業スコープ配下のコードをディレクトリ単位でざっと確認する。
3. 見つかった問題を重大度別(HIGH/MEDIUM/LOW)に整理する。
4. `.claude/plans/architecture-report-YYYYMMDD.md` にレポートとして保存する。
5. ユーザーに、対応するなら Planner に渡す準備がある項目を伝える。

## 注意
- このスキル自体はコードを変更しない(レポートのみ)。実際の改善は
  ユーザーが選んだ項目を Planner に渡してから行う。
- 研究の実験スクリプトは書き捨てられることも多いため、明らかに使い捨て
  用途と分かるコードは指摘の優先度を下げる。
