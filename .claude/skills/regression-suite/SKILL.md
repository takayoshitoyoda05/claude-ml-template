# 回帰テストスイート生成

Generatorの実装後、evaluatorが確認する「計画に書かれた検証項目」だけでは
見逃す可能性がある、計画外の意図しない挙動変化を広く検出する。

## 使いどころ
- 重要な変更(公開予定の実験結果に関わる、複数モジュールに影響する)のときに
  明示的に呼び出す。すべての変更に毎回使うとコストが増えすぎるため自動実行しない。
- tdd スキル(新規機能を1つずつ事前にテストする)や evaluator(計画の数値照合)
  とは役割が異なる。この skill は「実装後に、影響範囲全体を洗い出して広く固める」
  ことに専念する。

## 進め方
1. `git diff` で変更されたファイル・関数を特定する。
2. 変更された関数それぞれについて、入出力の境界値・典型値・エッジケース
   (空入力、極端な値、既存バージョンとの比較など)を洗い出す。
3. 既存のテストがあれば実行し、壊れていないか確認する。
4. テストが無い関数には新しくテストコードを書く。
5. すべて実行し、PASS/FAIL数と新規追加したテスト数を集計してレポートする。
6. 明らかに使い捨て用途と分かる実験スクリプト(1回限りの可視化スクリプトなど)は
   対象から除外してよい。判断に迷ったらユーザーに確認する。

## テストの永続化
生成したテストは使い捨てにせず、作業スコープ配下の tests/ に保存し、
将来の回帰テスト資産として積み上げる。

## 重要なルール
- ここで生成したテストの結果は、evaluator の「計画の期待値」による判定を
  上書きしない。数値の正しさの最終判断は必ず evaluator が行う。
- 現在の(バグを含むかもしれない)挙動をそのまま「正解」として固定しない。
  疑わしい挙動があれば、テストをPASSさせず指摘としてユーザーに報告する。

## 注意
- スコープ外のファイルを編集しない。CLAUDE_WORK_SCOPE に従う。
- コストが増えるため、明示的な呼び出し時のみ使う。
