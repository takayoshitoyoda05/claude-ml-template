# 設計インタビュー

ユーザーの設計書・計画を、実装に入る前に徹底的に問い詰め、認識のズレをなくす。

## 進め方
1. 作業スコープ直下に CONTEXT.md があれば先に読み、用語の解釈をそれに合わせる。
2. 対象の設計書(通常は docs/drafts/ 配下)を読み込む。指定がなければ、直近の
   会話や docs/drafts/ の中から対象を確認する。
3. 決定木を1枝ずつ辿る。各分岐で、未確定な判断を1つ特定する。
4. 質問は必ず1問ずつ出す。複数の質問を同時に出さない(ユーザーが混乱するため)。
5. 各質問には、こちらの推奨案を添える(例:「Aが良いと考えます、理由は〜」)。
6. コードや既存ドキュメントを読めば答えが分かる質問は、ユーザーに聞かず
   先に調査する。
7. ユーザーの回答を待ってから次の質問に進む。
8. すべての分岐が解消されたら、決定内容を設計書(docs/drafts/配下)に反映して
   更新する。
9. 最後に、固まった設計書を docs/active/ に移動する準備が整ったことを
   ユーザーに伝える(実際の移動は Planner が /ml-pipeline 実行時に行う)。

## 対象範囲
コードの設計に限らず、論文の論理構成、スライドの構成、実験計画など、
意思決定が必要な文書全般に使ってよい。ML/コードに限定しない。

## 注意
- スコープ外のファイルを編集しない。作業対象のプロジェクトディレクトリで
  claude を起動してから使う(CLAUDE_WORK_SCOPE が設定されていればそれに従う)。
