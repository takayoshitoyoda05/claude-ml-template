# 設計インタビュー

ユーザーの設計書・計画を、実装に入る前に徹底的に問い詰め、認識のズレをなくす。

## 進め方
**大原則(no-guess)**: 設計書の空欄・曖昧箇所を、もっともらしい内容で
勝手に埋めない。埋める内容は必ずユーザーへの質問から得る。
ユーザーが「お任せする」と明示した場合のみ、推奨案を提示して確認を取ってから
記入する(確認なしの記入は禁止)。

1. 作業スコープ直下に CONTEXT.md があれば先に読み、用語の解釈をそれに合わせる。
2. 対象の設計書(通常は docs/drafts/ 配下)を読み込む。指定がなければ、直近の
   会話や docs/drafts/ の中から対象を確認する。
3. 決定木を1枝ずつ辿る。各分岐で、未確定な判断を1つ特定する。
4. 未確定な判断を探すとき、以下の曖昧性タクソノミーを網羅の基準にする。
   全カテゴリについて「設計書に答えが書いてあるか」を確認し、
   書かれていないカテゴリから質問を作る。

   | カテゴリ | 必ず確認すること |
   |----------|-----------------|
   | 境界値 | 空・ゼロ・最大・重複・NaN/Inf の入力のときの挙動 |
   | 例外系 | 処理が失敗したとき何をするか(リトライ / スキップ / 停止 / ログのみ) |
   | 状態・順序 | 途中で中断されたらどうなるか。2回実行されたら(冪等性)。実行順序の前提 |
   | データ | 入力の形状・型・値域・欠損の扱い。出力の形式 |
   | 非機能 | 速度・メモリの許容値。「速く」「軽く」は必ず数値化する |
   | スコープ境界 | やらないこと(non-goals)。この設計書が解決しない問題の明文化 |

   研究系の設計書(実験・モデル変更を含む)では、さらに以下も確認する。

   | カテゴリ | 必ず確認すること |
   |----------|-----------------|
   | 仮説 | この変更で何がどう良くなると考えているか(1文) |
   | 変数 | 変えるもの(独立変数)と測るもの(従属変数) |
   | ベースライン | 何と比較するか。比較条件は揃っているか |
   | 成功基準 | 数値でいくつ以上/以下なら成功か(例: MAE がベースライン比 0.5 以上改善) |
   | 再現性 | seed・データ分割・環境の固定方法 |

   設計書の規模上あきらかに該当しないカテゴリは「(N/A: 理由)」と記録して次へ進んで
   よい(全カテゴリへの機械的な質問はしない。spec-checklist の規模に応じた判定と同じ基準)。
5. 質問は必ず1問ずつ出す。複数の質問を同時に出さない。
6. 各質問には、こちらの推奨案を添える(例:「Aが良いと考えます、理由は〜」)。
7. コードや既存ドキュメントを読めば答えが分かる質問は、ユーザーに聞かず
   先に調査する。
8. ユーザーの回答を待ってから次の質問に進む。
9. すべての分岐が解消されたら、決定内容を設計書(docs/drafts/配下)に反映して
   更新する。このとき「## 受け入れ条件」セクションを必須で生成する(無ければ新規作成、
   あれば決定事項を反映して更新)。全ての決定を、以下の6列を持つ Markdown テーブルの
   検証可能な要件に変換する(このテーブルなしでは Planner が計画作成を差し戻す)。

   | 列 | 意味 |
   |---|---|
   | ID | `R-001` 形式の連番。設計書内で一意・欠番なし |
   | 要件 | 検証可能な主張1つ(1行1要件。複合要件は分割する) |
   | 検証方法 | 実行コマンド(auto)または「(目視)」(manual) |
   | 期待結果 | `exit 0` / 数値条件 / 「人間承認」など、機械照合できる形式 |
   | 種別 | `auto` / `manual` |
   | 対象 | カバレッジ確認する対象モジュール(auto のみ、任意) |

   要件は可能な限り EARS 記法で書く。EARS は要件文を5つの型に当てはめる書式で、
   曖昧な要件を構造的に書けなくする。

   | 型 | 書式 | 例 |
   |----|------|-----|
   | 遍在(常時) | システムは<機能>しなければならない | システムは全ての乱数に seed を使用しなければならない |
   | イベント駆動 | <トリガー>のとき、システムは<応答>しなければならない | bag の読み込みが完了したとき、システムは特徴量を [0,1] に正規化しなければならない |
   | 状態駆動 | <状態>の間、システムは<応答>しなければならない | 学習中の間、システムは 1 epoch ごとに検証損失を記録しなければならない |
   | 望まない事象 | もし<条件>なら、システムは<応答>しなければならない | もし入力に NaN が含まれるなら、システムは該当 bag をスキップし WARNING を出力しなければならない |
   | オプション | <機能が有効>の場合、システムは<応答>しなければならない | 可視化フラグが有効の場合、システムは attention マップを outputs/ に保存しなければならない |

   1つの要件文に「かつ」「または」で複数の主張を入れない(分割して別IDにする)。

   曖昧で検証コマンド化できない決定は、この段階で追加の質問に分解し具体化する。

## 呼び出しコンテキストによる分岐
- /ml-pipeline の spec-checklist ゲート(手順3.3)から NEEDS_WORK で呼ばれた場合は
  **手順3・4・9(決定木の全枝走査・曖昧性タクソノミーの全カテゴリ確認・要件テーブルの
  全面再生成)は行わず**、指摘された箇所(MEDIUM以上)だけを対象に質問して解消する
  (全カテゴリの再インタビューはしない)。ただし**質問の作法(手順5〜8: 1問ずつ・
  推奨案を添える・コードで分かることは先に調査する・回答を待ってから次に進む)は、
  この限定モードでも変わらず適用する**。
  - 設計書がある場合: 設計書を更新して ml-pipeline に制御を返す。
  - 設計書が無い場合: 解消した回答を ml-pipeline に返し、
    planner の計画修正に使わせる。
- 単独で呼ばれた場合は、手順1〜9 の通常フローを実行し、完了時に「/ml-pipeline に渡すと
  spec-checklist で自動検査されます」と案内する。

## 知識の自動スタック(手順9の直後に必ず実施する)
設計書を更新したら、その内容から以下の3種を検出し、該当ファイルに追記する。
検出ゼロなら追記もゼロで構わないが、「検出したかどうか」の確認自体は必ず行う。

### (a) 新しい用語・略語 → CONTEXT.md
- 設計書に登場した独自の略語・固有名詞・プロジェクト特有の用語で、
  CONTEXT.md にまだ載っていないものを検出する。
- 検出したら、作業スコープ直下の CONTEXT.md の「用語一覧」表に追記する。
  CONTEXT.md が無ければ「# CONTEXT.md — ドメイン用語集」の見出しと
  「| 用語 | 定義 |」の2列テーブルだけの最小構成で新規作成する。
- 一時的な変数名や実装詳細(関数名など)は含めない。
  概念として繰り返し使われるものだけを追記する。

### (b) 重要な設計判断 → adr スキル
- トレードオフを伴う決定が2つ以上解消された場合、adr スキルを必ず使って
  docs/adr/ に記録する。「提案」ではなく、必ず実行すること。
- 決定が1つだけの場合はユーザーに確認してから adr を使う。
- 些細な決定(命名・書式の好みなど)は対象外。

### (c) 実験の再現条件 → docs/EXPERIMENT_LOG.md
- 設計書に具体的な数値・シード値・データセット名・モデル重みのパス等が
  含まれている場合、EXPERIMENT_LOG.md の該当エントリ(未作成なら新規)に
  「予定される実験条件」として追記する。
- 「今後決める」レベルの記述は追記しない。確定値のみを対象とする。

## 完了時
10. 固まった設計書を docs/active/ に移動する準備が整ったことをユーザーに
   伝える(実際の移動は Planner が /ml-pipeline 実行時に行う)。

## 対象範囲
コードの設計に限らず、論文の論理構成、スライドの構成、実験計画など、
意思決定が必要な文書全般に使ってよい。ML/コードに限定しない。

## 注意
- スコープ外のファイルを編集しない。CLAUDE_WORK_SCOPE に従う。
- 知識の自動スタックで CONTEXT.md が肥大化しないよう、繰り返し使われる
  概念だけを厳選する。判断に迷ったら追記しない。
