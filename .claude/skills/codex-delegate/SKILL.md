# Codex への実装委譲

レビュー(cross-review)とは別レイヤーの、実装作業の委譲。
Codex は独立タスクの量産・CLI 中心の操作・長時間の自律実行に強い。
設計判断が絡む作業は委譲せず Claude(generator)が担当する。

## 委譲に向くタスク / 向かないタスク

| 向く | 向かない |
|------|---------|
| 既存パターンに沿ったテストの量産 | 設計判断を伴う新機能 |
| 定型的な一括変換(フォーマット統一、リネーム) | ドメイン知識(MIL等)が必要な実装 |
| CLI操作中心の作業 | 複数モジュールにまたがるリファクタ |
| Claudeのレート制限中の退避作業 | 計画(.claude/plans/)に基づく実装(それはgeneratorの仕事) |

## 委譲の規律(全て必須)

Codex への依頼文には必ず以下を含める。「呼べること」と「狙った作業だけを
任せられること」は別であり、範囲を依頼文で固定する。

1. **cwd の明示**: 作業ディレクトリの絶対パスを依頼文に書く
2. **対象ファイルの明示**: 触ってよいファイル・ディレクトリを列挙する
3. **触らない範囲の明示**: 「上記以外のファイルは読んでよいが変更しない」と書く
4. **git 操作の禁止**: 「git add / commit / push は行わない」と必ず書く
   (コミットは委譲結果を確認した Claude 側が行う)
5. **完了条件の明示**: 何ができたら完了か(例: 全テストがPASSする)を書く

## 進め方

0. 委譲は作業ツリーがクリーンな状態で開始する。クリーンでなければ先に
   コミットまたは退避する(巻き戻しが「委譲の取り消し」と正確に一致する
   ことを保証するため)。
1. タスクが委譲に向くか上の表で判断する。向かない場合はその旨を伝え、
   通常の generator での実装を提案する。
2. 依頼文を組み立てる(規律1〜5を全て含める)。
3. Codex を呼び出す。
   - MCP(codex)が接続されていれば MCP 経由で依頼する
   - 接続されていなければ `codex exec "<依頼文>"` で実行する
4. Codex の作業完了後、変更内容を `git diff` で確認する。
   - 触らない範囲に変更が無いか必ず確認する
   - 範囲外の変更があれば `git checkout -- <ファイル>` で巻き戻す
     (開始時クリーンが前提なので、この巻き戻しは委譲による変更の取り消しと
     正確に一致する)
5. 問題なければ Claude 側で規約に沿ってコミットする
   (`<type>(delegate N): <要約>` 形式。N は関連する計画のステップ番号。
   計画に紐づかない単発の委譲なら 1)。
6. 委譲した作業も evaluator / evaluator-standards のレビュー対象になる
   (/ml-pipeline 内で使った場合)。委譲だからといって品質ゲートを免除しない。

## 依頼文のテンプレート

```
作業ディレクトリ: <絶対パス>
対象: <触ってよいファイル・ディレクトリの列挙>
制約:
- 上記以外のファイルは読んでよいが変更しない
- git add / commit / push は行わない
- <プロジェクト固有の制約(あれば)>
タスク: <具体的な作業内容>
完了条件: <何ができたら完了か>
```

## 注意
- guard_scope.py / guard_bash.py は Claude 側のフックであり、
  Codex の動作には効かない。だからこそ依頼文での範囲固定が必須
- 委譲結果を無確認でコミットしない。diff 確認は省略不可
