# 0002. manual要件の人間承認は保護パス化した承認ファイルで記録する

日付: 2026-07-04
状態: 承認

## 背景
自動検証できない要件(ドキュメント記載、見た目の確認など)は人間承認が必要になる。
承認の記録方法として、Claude がチャット上の承認発言を判定ファイルに転記する案が
自然だが、それでは Claude が勘違い・先走りで「承認済み」を書けてしまい、
物理ガードにならない。

## 検討した選択肢
1. Claude がチャット承認を判定ファイルに記録する(UXは楽、偽装・誤記入が可能)
2. 承認ファイル(.claude/spec/approvals.txt)を PROTECTED_PATH_PATTERNS に加え、
   ユーザーが `! uv run python .claude/hooks/spec_approve.py R-XXX` で追記する
3. manual要件そのものを禁止し全要件を auto に強制する

## 決定
選択肢2。承認ファイルへの Claude 経由の書き込み(Edit/Write・リダイレクト・tee)は
guard 系フックがブロックし、ユーザーのシェル実行(`!` プレフィックス)のみが
承認を記録できる。

## 理由
- 「人間が承認した」という事実を Claude が偽造・誤記入できない構造にするのが
  spec-compliance の趣旨(ガチガチ)と整合する
- 既存の保護パス機構(ガード自己書き換え防止)の自然な拡張で、実装コストが小さい
- 選択肢3は目視が本質的に必要な要件を設計書から追い出し、かえって漏れを生む

## トレードオフ
- 承認のたびにユーザーがコマンドを1回叩く手間が発生する
- approvals.txt はローカルファイルのため、別マシンでは承認が引き継がれない
  (CI では承認済み確認が必要なため、リポジトリにコミットする運用も検討余地がある)
