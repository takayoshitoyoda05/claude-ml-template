# 0001. spec_gate の判定は「verdict必須+抽出再実行」の二段構えにする

日付: 2026-07-04
状態: 承認

## 背景
spec-compliance(設計書適合チェック)の Stop ゲートが、auto要件の達成を
何を根拠に判定するかを決める必要があった。LLM(evaluator)の自己申告を
そのまま信じると「実行したつもりで PASS と書く」リスクが残り、
逆に全件をフックが再実行すると重いテストが Stop のたびに二重実行される。

## 検討した選択肢
1. フックが全 auto要件の検証コマンドを自ら再実行(決定的だが実行コスト最大)
2. evaluator の判定ファイル(verdict)を検査するのみ(軽いが LLM 申告を信用する)
3. verdict を必須としつつ、フックがランダム抽出N件を再実行して突き合わせ

## 決定
選択肢3。抽出件数は環境変数 CLAUDE_SPEC_RECHECK_N(既定3)で調整でき、
`all` 指定で選択肢1相当(全件再実行)に倒せる。

## 理由
- verdict の網羅チェック(全要件IDに判定があるか)で「漏れ」は決定的に検出できる
- 抜き打ち再実行があることで evaluator 側に「実行せず PASS と書く」誘因が働きにくい
- 重要な変更では RECHECK_N=all にすればガチガチ側へ、日常では既定値でコスト側へ、
  運用で振れる
- CI(spec_gate --ci)では全件再実行するため、最終防衛線は選択肢1と同等になる

## トレードオフ
- ローカルの既定運用では「抽出されなかった要件の実測」は evaluator の申告に依存する
  (CI で回収されるが、push まで検出が遅延する)
- 乱数抽出のため、同じ状態でも Stop のたびに検査対象が変わり、実行時間が読みにくい
