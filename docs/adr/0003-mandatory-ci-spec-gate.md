# 0003. spec-gate の CI ゲートを必須にする(claude-init が自動配置)

日付: 2026-07-04
状態: 承認

## 背景
Stop フックのゲートはセッション内の防衛線であり、セッション外の手動コミット、
別マシンからの push、フック未設定の環境には効かない。
「何の漏れもなく」という要件に対し、最終防衛線をどこに置くかを決める必要があった。

## 検討した選択肢
1. CIゲートなし(ローカル Stop ゲートのみ)
2. workflow テンプレを同梱し、使うかはプロジェクト側の任意
3. CIゲート必須: claude-init が .github/workflows/spec-gate.yml を自動配置

## 決定
選択肢3。claude-init / claude-update が spec-gate.yml を配置し、
PR 時に spec_gate --ci(auto要件の全件再実行+承認・監査の網羅確認)が走る。

## 理由
- ローカルゲートはすり抜け経路(セッション外変更)が構造的に残るため、
  リポジトリ側のゲートがないと「漏れなく」を満たせない
- CI では全件再実行するため、ローカルの抽出再実行(ADR-0001)の残余リスクを回収できる
- テンプレの既存 CI(verify-hooks)と同じ思想で一貫する

## トレードオフ
- GitHub Actions を使わない/プライベートな環境ではワークフローが死蔵される
  (動かないだけで害はないが、ノイズにはなる)
- CI 実行時間とコストが増える(重い評価コマンドを持つプロジェクトでは顕著)
- docs/active/ に設計書を置いたまま放置すると、無関係な PR も落ち続ける運用リスク
